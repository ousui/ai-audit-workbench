# Changelog

本文件记录 AI 审计工作台自身的版本变化。

普通项目审计结果、业务项目漏洞、整改记录、交付报告不写入本文件。

所有影响工作台执行流程、目录结构、状态定义、输出格式、Prompt、Schema、脚本行为、工具矩阵或报告模板的变更，都必须更新本文件。

---

## [0.1.0] - 2026-07-04

### Added

- 初始化 AI 通用审计工作台设计基线。
- 明确工作台不绑定单一 AI 工具，可适配 Codex、Claude Code、Cursor 或其他 Agent。
- 明确当前阶段主要面向静态代码审计和本地交付，但工作台长期能力不局限于静态分析。
- 明确长期能力边界：
  - SAST：静态代码分析
  - DAST：动态测试
  - REVERSE：逆向分析 / 构建产物分析
  - MANUAL：人工复核
  - CHK：整改复核
  - DELIVERY：交付生成
- 明确当前生产阶段默认只启用：
  - 审计地图生成
  - 静态代码分析
  - 本地工具扫描
  - AI 辅助候选问题判断
  - AI 辅助业务逻辑分析
  - 本地报告生成
  - 本地整改追踪表生成
- 明确公司 SOP 作为审计内容、阶段和检查项的重要参考来源，不直接作为 AI 执行协议。
- 明确工作台第一阶段会吸收 SOP 中的审计前准备、工具辅助、审计地图、静态分析和报告交付相关内容。
- 明确动态测试、移动端逆向、人工复核、AI 代码占比、CI/CD 持续化等 SOP 阶段暂不作为第一阶段默认执行流程，但需要保留长期扩展位置。
- 明确“审计地图”是正式审计前置产物，用于记录项目画像、技术栈、入口、配置、依赖、鉴权、数据访问层、高风险业务模块、覆盖范围和未覆盖原因。
- 明确环境检测 env-check 是工作台第 0 阶段能力，用于检测审计人员本机工具、SDK、多版本环境、工具路径和缺失影响。
- 明确 env-check 拆分为：
  - 全局环境检测：低频执行，记录审计人员电脑具备哪些能力
  - 单项目工具计划：每个项目执行，根据技术栈从工具矩阵中选择实际需要的工具
- 明确工具矩阵需要区分：
  - core：基础必备工具，例如 git、rg、python3
  - common_static：通用静态审计工具，例如 semgrep、gitleaks、trivy
  - sdk：技术栈 SDK，例如 go、java、node、php、flutter
  - security：语言或栈相关安全工具，例如 govulncheck、dependency-check、retire.js
  - dast：动态测试工具
  - reverse：逆向分析工具
- 明确工具缺失处理原则：
  - 缺少 core 工具时阻断工作台启动
  - 缺少当前技术栈必备 SDK 时阻断对应项目相关扫描
  - 缺少推荐工具时记录 blocked 或 skipped，不得写成无风险
  - 缺少 DAST / REVERSE 工具时，不影响当前 SAST-only 阶段
- 明确推荐支持多版本开发环境管理工具，例如 mise、asdf、sdkman、nvm、pyenv 等。
- 明确工具安装位置需要记录：
  - system
  - user
  - sandbox
  - version manager
  - active version
  - resolved path
  - discovered instances
- 明确沙箱和授权边界：
  - 工具默认只读被审项目源码
  - 工具输出只能写入 run 目录或缓存目录
  - 不得默认写入被审项目源码
  - 联网、动态测试、启动服务、逆向样本处理、上传或写回外部系统必须单独确认
- 明确工作台核心自动化原则：
  - 程序优先：可标准化、可复跑、可校验的任务优先交给程序
  - 证据优先：AI 默认基于程序生成的证据包和候选池判断
  - 受控探索：高风险或低覆盖场景允许 AI 深度探索，但探索结果必须先转成候选项
  - 分级自治：AI 可在策略允许范围内自动采信低风险高置信结果，人工门禁只拦高风险、低置信、边界变更和流程变更
- 明确三层执行模型：
  - Script / Program：负责确定性执行、收集、转换、校验
  - AI / Agent：负责候选判断、语义分析、业务逻辑推理
  - Human Gate：负责边界确认、最终采信、流程变更和高风险定级
- 明确快速扫描与深度探索需要通过审计模式区分：
  - FAST_STATIC：快速静态模式，AI 默认只基于证据包和候选池判断
  - STANDARD_STATIC：标准静态模式，AI 可请求补充上下文
  - DEEP_STATIC_EXPLORE：深度静态探索模式，AI 可在授权范围内只读探索仓库，但探索结果必须回写为候选项
- 明确 Human Gate 不是固定人工全拦，而是可配置风险门禁。
- 明确后续需要设计 autonomy policy，用于控制哪些结论可以自动采信，哪些必须人工确认。
- 明确任何正式问题都必须经过 candidate → triage → merge → delivery 流程，不允许 AI 自由探索后直接写入业务报告。
- 明确报告生成阶段不得重新审计，不得新增、删除、重排或改变问题等级。
- 明确交付阶段只能基于已合并结果渲染报告和整改追踪表。
- 增加 debug 模式设计，作为工作台流程观察、留痕、调试和复盘机制；debug 产物仅用于工作台改进，不进入业务报告，不扩大工具权限，不替代审计结论。
- 增加 core 工具安装指南，覆盖 git、ripgrep、python3、bash、find、grep、sed、awk、jq、tar，并明确 macOS、Linux、Windows 的基础安装建议与缺失处理策略。
- 增加 env-check 结果 JSON Schema，用于约束本机环境检测结果结构，覆盖主机信息、工具检测摘要、工具实例、多版本 SDK、缺失影响和权限边界。
- 增加 env-check 示例结果文件，用于展示本机环境检测输出结构、工具状态、路径脱敏、多版本实例、缺失工具影响和权限边界。
- 增加 core env-check 检测脚本，用于检测基础工具可用性、版本、路径、多实例、缺失影响和权限边界，并输出本地环境检测结果文件。

### Notes

- 当前版本仅建立工作台设计基线和变更记录。
- 当前版本尚未创建目录结构、环境检测脚本、工具矩阵、Prompt、Schema 或报告模板。
- 后续步骤应按小步提交方式推进，避免一次性生成复杂流程。
