# Workbench Change Policy

本文件定义 AI Audit Workbench 的变更规则。

本文件的目标是防止 AI 工具在迭代过程中无边界扩展、重复定义规则、破坏产物稳定性或污染生产流程。

## 1. 基本原则

工作台鼓励自动化，但自动化必须受控。

核心原则：

能程序化的，不交给 AI 自由发挥。  
需要语义判断的，不强行程序化。  
影响边界和结论的，必须进入门禁。

## 2. 默认禁止 AI 修改的内容

除非用户明确要求“修改工作台流程”，否则 AI 不得修改：

- AGENTS.md
- CHANGE_POLICY.md
- CHANGELOG.md
- README.md
- config/
- env/
- prompts/
- schemas/
- rules/
- scripts/
- templates/

以上内容属于工作台流程、规则、模板或程序资产。

AI 执行普通审计任务时不得自行修改。

## 3. 允许 AI 默认写入的内容

执行具体审计任务时，AI 只允许写入：

- runs/
- tmp/
- delivery/
- 当前用户明确指定的输出目录

AI 不得修改被审项目源码。

AI 不得修改历史 run 目录中的结果。

AI 不得在未经确认的情况下覆盖既有产物。

## 4. 工作台流程变更要求

如果用户明确要求修改工作台流程，AI 必须先输出变更方案。

变更方案必须包含：

1. 变更目标
2. 修改文件列表
3. 影响范围
4. 回滚方式
5. 验证方式
6. 是否影响历史审计产物
7. 是否需要更新 CHANGELOG.md
8. 是否影响审计结论状态
9. 是否影响报告模板
10. 是否影响工具执行边界

未经用户确认，不得直接修改流程。

## 5. CHANGELOG 要求

任何影响以下内容的变更，都必须更新 CHANGELOG.md：

- 工作台执行流程
- 目录结构
- 审计模式
- 状态定义
- 输出格式
- Prompt
- Schema
- 脚本行为
- 工具矩阵
- 报告模板
- 交付模板
- 自动采信策略
- Human Gate 策略

普通项目审计结果、业务项目漏洞、整改记录、交付报告不写入 CHANGELOG.md。

## 6. 程序化变更边界

工作台鼓励将流程化、标准化、可复跑的任务脚本化。

适合程序化的任务包括：

- 环境检测
- 工具矩阵匹配
- 项目初始化
- Git 信息读取
- 技术栈识别
- 审计地图初稿生成
- 配置、依赖、路由、入口文件枚举
- 工具扫描执行
- 工具原始结果归档
- 候选问题生成
- 候选编号和去重
- Schema 校验
- 报告和整改表模板渲染
- 产物一致性校验
- 本机路径检查
- 敏感信息泄露检查

脚本不得承担以下职责：

- 自动放宽 FIND 成立标准
- 自动将 CAND 提升为 FIND
- 自动将 REVIEW / RUNTIME 写成已验证
- 自动改变风险等级
- 自动删除候选问题且不留记录
- 自动修改历史 run 结果
- 自动修改被审项目源码
- 自动执行动态测试
- 自动执行生产探测
- 自动执行逆向分析
- 自动上传报告
- 自动写回团队表格

## 7. AI 判断边界

AI 适合处理：

- 候选问题是否成立的判断
- 扫描命中的误报分析
- 调用链和数据流语义分析
- 鉴权、越权、资源归属校验分析
- 业务逻辑风险分析
- source-sink 可达性分析
- 反证检查
- 修复建议生成
- 报告文字润色

AI 不得：

- 绕过候选池直接新增正式问题
- 把工具扫描命中直接写成 FIND
- 在交付阶段重新审计
- 修改被审项目源码
- 把静态疑似写成动态验证结论
- 伪造未执行的审计阶段
- 为了增加问题数量而降低标准

## 8. 深度探索边界

工作台允许深度探索模式，但必须受控。

默认 FAST_STATIC 模式下，AI 只能基于证据包和候选池判断。

STANDARD_STATIC 模式下，AI 可以请求补充上下文，但补充上下文由程序读取并回填。

DEEP_STATIC_EXPLORE 模式下，AI 可以在授权范围内只读探索仓库。

但深度探索结果必须先转化为候选项，不得直接写入正式报告。

统一流程为：

AI explore  
→ new candidate  
→ triage  
→ merge  
→ delivery

## 9. Human Gate 策略

Human Gate 是可配置风险门禁，不是固定人工全拦。

工作台默认初始自治等级为 A1。

### A0：人工全确认

AI 只能建议，所有报告收录都需要人工确认。

### A1：低风险自动处理

AI / 程序可以自动处理：

- FP
- NO_RISK
- CAND 不进报告
- 低价值工具命中归档

人工重点查看：

- FIND
- REVIEW
- RUNTIME
- BLOCKED

### A2：中低风险自动采信

满足策略条件时，P2 / P3 高置信 FIND 可以自动进入报告。

### A3：高置信自动交付

经过足够样本验证的稳定规则可以自动交付。

例如：

- 明确硬编码密钥
- .env 被提交
- 明确高危依赖
- 明确测试代码误报
- 明确 localStorage 存 access_token

### A4：低风险项目全自动交付

仅适用于低风险项目、工具覆盖完整、无 P0/P1、无 BLOCKED、无 RUNTIME 且 validator 全通过的场景。

## 10. 必须人工确认的事项

以下事项必须人工确认：

- 审计范围
- 审计分支和 commit
- 是否允许联网
- 是否允许动态测试
- 是否允许启动服务
- 是否允许逆向分析
- 是否上传报告
- 是否写回外部系统
- 是否修改工作台流程
- P0 / Critical 风险定级
- 高风险业务逻辑结论
- 资金 / 支付 / 提现类高风险结论
- 改变风险等级标准
- 改变输出 Schema
- 改变报告模板
- 删除大量候选
- 采纳新的自动化策略

## 11. env-check 变更边界

env-check 是工作台第 0 阶段能力。

涉及以下内容的变更必须确认：

- core 工具定义
- 工具矩阵结构
- 工具缺失阻断策略
- SDK 多版本检测逻辑
- 安装建议
- 沙箱策略
- 联网策略
- 缓存目录
- 工具输出目录
- 本机路径脱敏规则

env-check 结果中可能包含本机路径，不得进入业务方报告。

## 12. Delivery 变更边界

交付阶段只能基于已合并结果渲染产物。

delivery 阶段不得：

- 重新审计
- 新增问题
- 删除问题
- 重排问题
- 改变风险等级
- 改变确认状态
- 引入新的候选
- 读取被审项目源码补结论

如需改变交付内容，必须先回到 merge 结果或模板变更流程。

## 13. 禁止行为

- 不得为了报告更好看而改变风险等级。
- 不得为了问题数量而放宽 FIND 标准。
- 不得把实验流程直接覆盖生产流程。
- 不得修改历史 run 目录中的结果。
- 不得让 delivery 阶段重新审计。
- 不得让交付阶段新增、删除、重排或改变问题等级。
- 不得把未执行的动态测试、逆向分析或人工复核写成已覆盖。
- 不得把缺失工具导致的未覆盖写成无风险。
- 不得把 AI 自由探索结果直接写入业务报告。
