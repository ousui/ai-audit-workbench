# AI Audit Workbench

AI Audit Workbench 是一个面向代码审计工作的 AI 辅助审计工作台。

本工作台不绑定单一 AI 工具。当前主要面向 Codex 使用，后续可适配 Claude Code、Cursor 或其他 Agent。

## 1. 工作台目标

本工作台用于帮助审计人员高效、稳定、可复跑地完成代码审计交付。

核心思想：

程序收集事实。  
AI 判断候选。  
程序合并结果。  
程序渲染报告。  
人工进行质量门禁。

本工作台不是业务开发平台，不用于修改业务源码，不用于替业务方修复漏洞。

## 2. 当前阶段目标

第一阶段重点建设高吞吐静态审计流程。

当前默认启用：

- 审计地图生成
- 静态代码分析
- 本地工具扫描
- AI 辅助候选问题判断
- AI 辅助业务逻辑分析
- 本地审计报告生成
- 本地整改追踪表生成

当前默认不启用：

- 动态测试
- 生产环境探测
- 启动被审项目服务
- 移动端逆向 / 构建产物逆向
- 自动上传报告
- 自动写回团队项目表格
- 修改被审项目源码

## 3. 长期能力规划

工作台长期支持以下阶段：

| 阶段 | 含义 |
|---|---|
| SAST | 静态代码分析 |
| DAST | 动态测试 |
| REVERSE | 逆向分析 / 构建产物分析 |
| MANUAL | 人工复核 |
| CHK | 整改复核 |
| DELIVERY | 交付生成 |

当前第一阶段只实现 SAST 和 DELIVERY 的最小闭环。

未启用的阶段必须明确记录为 skipped、blocked 或 pending，不得伪造成已覆盖。

## 4. SOP 定位

公司 SOP 是审计内容、阶段和检查项的重要参考来源。

工作台负责把 SOP 中适合当前阶段的内容转化为：

- 可执行流程
- 可复跑脚本
- 可校验产物
- 可追溯证据
- 可交付报告

SOP 不直接作为 AI 执行协议。

AI 执行时必须优先遵守：

1. 当前用户指令
2. 工作台规则
3. 状态定义
4. 目录边界
5. 输出要求

## 5. 核心设计原则

### 5.1 程序优先

可标准化、可复跑、可校验的任务优先交给程序或脚本。

例如：

- 环境检测
- 工具矩阵匹配
- Git 信息读取
- 技术栈识别
- 审计地图初稿生成
- 工具扫描执行
- 候选问题生成
- JSON Schema 校验
- 报告模板渲染
- 敏感信息脱敏检查
- 产物一致性校验

### 5.2 证据优先

AI 默认基于程序生成的证据包和候选池进行判断。

AI 不应在默认模式下自由探索整个仓库。

### 5.3 受控探索

针对高风险或低覆盖场景，允许 AI 在授权范围内只读探索仓库。

但 AI 探索结果不得直接写入正式报告，必须先转化为候选项，并进入统一流程：

candidate → triage → merge → delivery

### 5.4 分级自治

Human Gate 不是固定人工全拦，而是可配置风险门禁。

在样本足够、规则稳定、置信度足够时，AI 可以在策略允许范围内自动采信低风险高置信结果。

人工重点处理：

- 高风险
- 低置信
- 业务语义不明
- 动态验证缺失
- 边界变更
- 流程变更
- P0 / Critical 定级

## 6. 审计模式

### FAST_STATIC

快速静态模式。

适合大量项目的批量初筛。

特点：

- 程序生成审计地图
- 程序生成候选池
- AI 默认只基于证据包和候选池判断
- 不自由探索仓库

### STANDARD_STATIC

标准静态模式。

适合普通核心项目。

特点：

- 以证据包和候选池为主
- AI 可以请求补充上下文
- 补充上下文由程序读取并回填
- AI 不直接绕过候选池生成正式问题

### DEEP_STATIC_EXPLORE

深度静态探索模式。

适合高风险项目或低覆盖项目。

特点：

- AI 可以在授权范围内只读探索仓库
- 可以发现程序规则未捕捉到的新候选
- 探索结果必须回写为候选项
- 不得直接写入正式报告

### DAST

动态测试模式。

当前阶段默认不启用。

### REVERSE

逆向分析 / 构建产物分析模式。

当前阶段默认不启用。

## 7. 环境检测

环境检测是工作台第 0 阶段能力。

目标是检测审计人员当前电脑是否具备审计条件，包括：

- 基础命令
- 本地工具
- 多版本 SDK
- 安全扫描器
- 动态测试工具
- 逆向分析工具
- 工具安装路径
- 当前激活版本
- 缺失影响
- 推荐安装方式

环境检测分为两层：

| 层级 | 说明 |
|---|---|
| 全局环境检测 | 低频执行，记录审计人员电脑具备哪些工具能力 |
| 单项目工具计划 | 每个项目执行，根据项目技术栈从工具矩阵中选择需要的工具 |

缺少 core 工具时阻断工作台启动。

缺少当前技术栈必备 SDK 时阻断对应项目相关扫描。

缺少推荐工具时记录 blocked 或 skipped，不得写成无风险。

## 8. 审计地图

每次正式审计必须建立审计地图。

审计地图用于：

- 记录项目画像
- 指导候选生成
- 指导 AI 判断重点
- 记录审计覆盖范围
- 生成报告中的项目概况和未覆盖说明

审计地图至少记录：

- 项目基本信息
- Git 分支和 commit
- 技术栈
- 目录结构
- 配置文件
- 依赖文件
- 路由 / API 入口
- 鉴权 / 权限相关模块
- 数据访问层
- 文件上传 / 下载模块
- 高风险业务模块
- 前端 API 调用和本地存储
- 排除范围
- 未覆盖原因

## 9. 典型产物流

第一阶段的目标产物流：

project source  
→ audit map  
→ evidence pack  
→ candidate pool  
→ AI triage  
→ merge result  
→ delivery report / tracking table  
→ validation result

## 10. 当前不承诺

当前工作台第一阶段不承诺：

- 漏洞完全准确
- 漏洞完全覆盖
- 动态可利用性验证
- 移动端逆向覆盖
- 自动整改闭环
- 在线上传
- 自动写回团队项目表格

第一阶段优先保证：

- 输入稳定
- 输出稳定
- 可复跑
- 可交付
- 可审计
- 可逐步扩展
